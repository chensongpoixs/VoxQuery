## 1. 项目概述维度

### Q1.1 用一句话介绍这个项目

**面试官意图分析**:

这道题是面试的开场破冰题，面试官有 3 层意图：

- **表层意图 — "你能说清楚自己做了什么吗"**：测试概括表达能力。如果给你 30 秒你都说不清楚这个项目是什么，面试官会认为你对项目理解不深，或者压根不是核心贡献者。
- **中层意图 — "你的项目定位清晰吗"**：判断你是"技术执行者"还是"有业务思维的人"。好的回答会先讲业务场景（企业 + 知识库）再讲技术方案，差的回答一上来就罗列用了什么模型、什么框架。
- **深层意图 — "你和其他候选人有什么不同"**：面试官一天可能面 5 个人，大部分项目都叫"RAG 问答系统"。你在第一句话里能不能抛出差异化点（私有化部署、语音交互、行业定制），决定了面试官后续会深入问你还是草草带过。

**高手回答结构**：业务场景（1句）→ 技术方案（1句）→ 核心差异（1句），三句话锁定印象。

**低级回答特征**：① 从头讲到尾像背书 ② 只用技术语言没有业务语言 ③ 说不出"为什么"只有"做了什么" ④ 没有差异化意识，听起来跟网上教程做的 DEMO 一样。

---

**分析流程**: 项目定位 → 核心功能 → 技术栈 → 差异化

**参考答案**:

> 这是一个面向企业的**企业内部知识库问答系统 + 语音对话助手**，采用 ASR → LLM+RAG → TTS 三阶段流水线架构，基于开源模型（Gemma-3 27B、BGE-M3、Whisper-large-v3、CosyVoice2）实现完全私有化部署，支撑员工通过自然语言或语音提问，快速检索企业内部的业务规范、设备手册、运维文档等私域知识。

**关键词**: 私有化部署、RAG、多模态、开源模型、企业

### Q1.2 这个项目解决了什么痛点？

**面试官意图分析**:

这道题是面试官测试"产品思维"的核心题，背后的逻辑链是：

- **他想干什么 — 判断你是否真的理解"为什么要做这个项目"**：很多候选人能把项目讲得很技术，但一问到"为什么要做"就语焉不详。面试官想通过这道题区分"照着 PRD 写代码的人"和"理解业务价值驱动技术选型的人"。
- **考察要点 ① — 你是否做过真实的用户调研**：如果你说的痛点都是"提高效率"、"降低成本"这种万金油，说明你没接触过真实用户。反之，如果你能说出"巡检人员双手操作设备不便打字"、"老专家退休知识断层"这种行业特有痛点，说明你真的深入过业务场景。
- **考察要点 ② — 技术方案是否对应真实痛点**：面试官会暗中对比你前面介绍的技术方案和现在说的痛点是否一致。如果你说痛点是企业数据安全，但技术方案里全是云服务，这就自相矛盾了。
- **考察要点 ③ — 能否量化**：你能不能在痛点里加入数据（如"翻阅数百页手册耗时 10-30 分钟"），决定了你的可信度。

**低级回答特征**：① 痛点泛泛而谈，放在任何项目都适用 ② 只说"现有方案不好"但不分析为什么不满足需求 ③ 没有行业特殊性，听起来像 ChatGPT 都能解决。

---

**分析流程**: 业务场景 → 痛点 → 解决方案 → 量化效果

**参考答案**:

| 痛点 | 现状 | 解决方案 |
|------|------|---------|
| **知识检索效率低** | 工程师需翻阅数百页 PDF 手册找参数，耗时 10-30 分钟 | RAG 语义检索 + LLM 精准回答，秒级响应 |
| **经验依赖严重** | 老专家退休后知识断层，新人学习成本高 | 知识库沉淀企业文档，AI 7×24 在线解答 |
| **语音场景需求** | 巡检人员双手操作设备，不便打字 | 语音对话全链路，边说边查 |
| **数据安全顾虑** | 飞书/钉钉等 SaaS 产品的 AI 助手会把数据上传云端 | 完全私有化部署，数据不出企业 |
| **行业术语壁垒** | 通用搜索引擎不理解"主变"、"PT"、"刀闸"等行业术语 | 100+ 术语词典 + 50+ 同义词组 |

### Q1.3 和飞书智能伙伴、钉钉 AI 助手有什么不同？

**面试官意图分析**:

这是一个高风险问题，面试官在测试 3 件事：

- **他想干什么 — 深水区压力测试**：面试官直接拿行业头部产品来"怼"你，本质上是在追问"你做的这个东西，在公司内部到底有没有必要自研？"如果你回答不好，面试官会认为你做的是重复造轮子，或者是技术自嗨。
- **考察要点 ① — 竞品调研深度**：你研究过飞书/钉钉 AI 的实现原理吗？你知道它们的数据处理流程吗？如果你能准确说出"飞书 AI 助手需要把文档上传到飞书云端做索引"，说明你真的做了功课。
- **考察要点 ② — 是否理解"私有化部署"的本质差异**：这道题的关键不在技术优劣，而在**数据主权**。企业（尤其是央企国企）有合规红线，数据不得离开企业机房。这是 SaaS 方案的不可逾越壁垒，也是你方案的根本立足点。
- **考察要点 ③ — 是否一踩一捧**：面试官想看你能否客观评价竞品。如果你说"飞书技术不行"，会显得不专业。正确姿势是"飞书AI是好产品，但场景不同——它是通用SaaS，我们是私有化+行业深度定制，不构成直接竞争，而是解决不同需求"。

**回答陷阱**：
- ❌ 贬低竞品："飞书AI不够智能"——显得不客观
- ❌ 技术优越感："我们的开源模型比闭源模型好"——开源和闭源各有优劣
- ❌ 回避问题："定位不同不好比较"——显得没有思考过
- ✅ 正确切入："数据主权是不可绕过的硬约束，这一点决定了技术路线根本不同"

**关键论述**: "飞书的 AI 助手是通用 SaaS，数据要上传到字节的服务器。企业尤其央企国企，数据不出机房是硬性要求，我们的方案就是为此设计的。"

**参考答案**:

| 维度 | 飞书/钉钉 AI | 本系统 |
|------|-------------|--------|
| **数据存放** | 云端 SaaS，需要将企业文档上传至第三方服务器 | **完全私有化部署**，数据在企业内部 |
| **模型可控** | 黑盒闭源模型，无法定制 | **开源模型全可控**，可微调、量化、裁剪 |
| **行业适配** | 通用办公场景 | **企业深度定制**（术语词典、同义词、行业 Prompt） |
| **GPU 资源** | 不涉及 | **4×RTX 4090 精确绑定**，消费级显卡跑 27B 模型 |
| **语音能力** | 基础 | **全链路 ASR+LLM+TTS**，零样本音色克隆 |
| **合规审计** | 日志在云端 | **审计日志在本地**，满足行业合规要求 |

**关键论述**: "飞书的 AI 助手是通用 SaaS，数据要上传到字节的服务器。企业尤其央企国企，数据不出机房是硬性要求，我们的方案就是为此设计的。"

### Q1.4 你在项目中负责了什么？

**面试官意图分析**:

这道题的面试官意图非常明确，但回答要有技巧：

- **他想干什么 — 验证你简历上写的"负责系统架构设计"是不是真的**：面试官见过太多候选人把团队成果说成个人成果。通过这道题，他在看你能否精准地说出具体的模块、具体的决策、具体的技术细节。
- **考察要点 ① — 参与深度**：如果你的回答都是"我负责了后端开发"这种模糊描述，面试官会判定你是浅参与。如果你能说出"chunker 的 _adjust_boundary 方法保护了 4 种企业模式不被切断"，面试官就知道你确实写代码了。
- **考察要点 ② — 角色定位**：你是架构主导者还是模块实现者？这道题的措辞会暴露你的真实角色。如果你是架构主导者，你的回答会从"整体架构设计"讲起；如果你只是参与者，你会从自己做的具体模块讲起。
- **考察要点 ③ — 广度 vs 深度**：面试官通过你的回答策略判断你的定位——全栈还是专精。如果你把所有模块都列一遍，面试官会认为你是全栈开发；如果你深入展开 1-2 个模块的技术细节，面试官会认为你是该方向的专精。

**回答策略建议**：
- 如果是面全栈/架构岗：先覆盖广度（架构设计 + 各模块），再挑一个最有技术含量的深入讲
- 如果是面算法/RAG 岗：重点展开模块 1、4（文档处理 + RAG 检索），其他模块一带而过
- 如果是面推理部署岗：重点展开模块 5（LLM vLLM 部署），强调 GPU 显存编排和量化

**低级回答特征**: ① "我是后端开发，前端不了解"——这会暴露你对系统全景理解不够 ② 说的都是项目管理类工作，没有技术细节 ③ 用"我们"代替"我"——面试官关心的是"你"做了什么。

**参考答案（可分角色调整）**:

> 我负责了项目的**整体架构设计**和**全栈开发**，包括：
> - **系统架构设计**: 6 个微服务的解耦设计、API Gateway 路由调度、Docker Compose GPU 绑定编排
> - **RAG 核心链路**: 文档分段引擎（3 种策略）、语义检索（多路召回+Re-rank）、同义词映射、增量更新机制
> - **LLM 推理部署**: Gemma-3 27B 在 2×RTX 4090 上的 vLLM 张量并行部署、INT8 量化、KV Cache 优化
> - **对话管理**: 多轮记忆（Redis）、指代消解、流式 SSE、兜底策略
> - **语音全链路**: ASR（Whisper WebSocket 流式）+ TTS（CosyVoice2 音色克隆）的集成
> - **前端开发**: Next.js 全栈 UI，含对话界面、语音录音、Markdown 渲染、知识库管理

## 2. 系统架构维度

### Q2.1 为什么选择微服务架构而不是单体？

**面试官意图分析**:

这是架构设计能力的核心考题，面试官通过它判断你在技术决策上是否有"权衡思维"：

- **他想干什么 — 判断你是"跟风用微服务"还是"真正理解微服务的适用场景"**：微服务这些年被滥用了，很多项目为了微服务而微服务，徒增复杂度。面试官想看到你能否说出"我这个场景下微服务具体解决了什么单体解决不了的问题"。
- **考察要点 ① — 是否能结合具体约束来论证**：你的项目有 4 个 GPU、4 个模型，这是硬约束。如果你把 GPU 绑定作为微服务的核心理由，面试官会觉得你的架构决策是"问题驱动"的，不是"概念驱动"的。
- **考察要点 ② — 是否理解微服务的代价**：好的架构师一定会在论证中主动提 trade-off。如果你只说微服务好处不说代价，面试官会追问"那通信延迟怎么办？数据一致性怎么保证？"。自己先说代价并解释为什么可接受，是高段位回答。
- **考察要点 ③ — "技术栈异构"这个理由的含金量**：面试官知道 AI 领域的依赖地狱问题（PyTorch 版本冲突、CUDA 版本不兼容）。如果你能说出"ASR 用 faster-whisper 有特定 PyTorch 版本要求，TTS 用 CosyVoice2 需要特定依赖，放一起会有依赖冲突"，这证明你真正部署过模型。

**高级回答策略**：先讲约束（4 GPU + 4 模型 + 依赖冲突）→ 逐个解释微服务如何解决 → 主动提代价并解释为什么可接受。这个结构会让面试官觉得你有"架构师的思维框架"。

**参考答案**:

> 选择微服务架构的核心原因有以下几点：
>
> 1. **GPU 资源隔离需求**：4 个模型需要绑定到不同的 GPU，微服务架构可以让每个模型的 Docker 容器**精确绑定指定的 GPU**。单体架构做不到这种精细的 GPU 分配。
>
> 2. **独立扩缩容**：Embedding 服务在批量文档入库时会成为瓶颈，需要独立扩容。LLM 推理服务在高峰期压力大。微服务可以按需扩容各自的服务实例。
>
> 3. **故障隔离**：TTS 服务挂了不影响文本问答，ASR 服务 OOM 不影响 LLM 推理。单体架构一个模块崩掉整个服务不可用。
>
> 4. **技术栈异构**：ASR 用 faster-whisper（PyTorch），LLM 用 vLLM（有特定 CUDA 版本要求），TTS 用 CosyVoice2（有特定依赖），放在一起会有依赖冲突。
>
> **代价**：微服务增加了网络通信开销（内部 REST API 调用），但本项目的延迟瓶颈在 GPU 推理（秒级），网络延迟（毫秒级）可以忽略。

### Q2.2 为什么选择 FastAPI 作为 API Gateway？

**面试官意图分析**:

这道题看起来是技术选型题，实际上考察的是你是否理解"AI 项目网关"和"传统 Web 网关"的需求差异：

- **他想干什么 — 测试你是否能"看场景选技术"**：如果候选人直接说"FastAPI 性能好、文档自动生成"，面试官会觉得这像是看了官网介绍的水平。真正做过 AI 网关的人会说：AI 网关的核心需求是**异步等待长时间推理结果 + SSE 流式输出**，而 FastAPI 的 async/await + sse-starlette 恰好精准满足这个需求。
- **考察要点 ① — 是否理解 async/await 在本项目中的具体价值**：普通 CRUD 应用用同步就够了，但你的 Gateway 需要同时等待 LLM（1-3秒）、RAG（0.5秒）、ASR、TTS 的响应——如果同步阻塞，20 个并发用户就会明显延迟。async/await 让 Gateway 在等待 GPU 推理时不占用线程。
- **考察要点 ② — 选型分析是否结合了"Python AI 生态"这个前置约束**：面试官想看你是否能意识到——除了 FastAPI，其他语言的方案（Go gin / Node Express）在"调用 Python AI 服务客户端"这件事上天然不兼容。这是"场景特异性"分析，比泛泛的性能对比更有说服力。
- **考察要点 ③ — 是否了解 Python 异步的局限**：如果你能补充一句"Python GIL 限制了 CPU 密集操作的并发处理，但本项目中 Gateway 是纯 IO 等待（等下游 HTTP 响应），不涉及 CPU 密集计算，所以 async/await 的收益最大化"，面试官会觉得你对 Python 运行时的理解很深入。

**参考答案**:

| 维度 | FastAPI | Flask | Go(gin) | Node(Express) |
|------|---------|-------|---------|---------------|
| **异步性能** | ⭐⭐⭐⭐⭐ 原生 async/await | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **生态兼容** | ⭐⭐⭐⭐⭐ Python(AI 生态) | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐ |
| **SSE 流式** | ⭐⭐⭐⭐⭐ sse-starlette | ⭐⭐ | ⭐⭐⭐ | ⭐⭐⭐ |
| **自动文档** | ⭐⭐⭐⭐⭐ Swagger 自动生成 | ⭐⭐ | ⭐⭐ | ⭐⭐ |
| **WebSocket** | ⭐⭐⭐⭐⭐ 原生支持 | ⭐⭐ | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |

> 核心原因：
> 1. **async/await 原生支持**：Gateway 需要同时等待 LLM（1-3秒）、RAG（0.5秒）、ASR、TTS 的响应，FastAPI 的异步非阻塞特性可以高效处理并发。
> 2. **SSE 流式响应**：`sse-starlette` 库成熟稳定，实现打字机效果非常简单。
> 3. **Python 生态**：AI 相关 SDK（httpx、Redis、JWT）在 Python 生态最成熟。
> 4. **自动生成 API 文档**：`/docs` 路径自动生成 Swagger UI，联调方便。

### Q2.3 服务之间是怎么通信的？

**面试官意图分析**:

这道题面试官在考察你对分布式系统通信协议的"场景化决策能力"：

- **他想干什么 — 区分"只会用 HTTP 的人"和"知道什么时候不该用 HTTP 的人"**：如果你说"我只会 HTTP，别的没试过"，面试官会认为你的技术视野有限。如果你能分析 HTTP / gRPC / 消息队列各自的适用场景并给出具体理由，面试官会觉得你在工程决策上有经验。
- **考察要点 ① — "为什么不用 gRPC"的回答暴露技术深度**：gRPC 有更好的性能（Protobuf 序列化 + HTTP/2 多路复用），但面试官想看你是否知道 gRPC 在 Python 生态的适配成本。如果你说"gRPC 需要 .proto 文件、代码生成、调试工具链，在 Python AI 项目中性价比不高"，这证明你评估过。
- **考察要点 ② — "为什么不用消息队列"的回答暴露复杂度嗅觉**：好的工程师知道"不引入"和"引入"同样重要。如果你能说出"消息队列增加了 Broker 运维成本、消息丢失风险、最终一致性复杂度，而本项目 QPS < 20 不需要解耦到这种程度"，这说明你有"复杂度管控"意识。
- **考察要点 ③ — 容错机制是加分项**：即使面试官不问容错，你主动讲容错降级策略（LLM 挂了 → fallback，RAG 挂了 → 降级为纯 LLM），会让面试官觉得你在生产环境部署过系统。

**隐藏考点 — debugability（可调试性）**：在生产环境中，HTTP 的 curl 可调试性比 gRPC 的 grpcurl 好很多。如果面试官问"生产环境半夜出故障你怎么快速排查"，这个点就是关键。

**参考答案**:

> 所有服务通过 Docker 内部网络（`kb-net`）通信，使用 **HTTP REST API** 作为通信协议。
>
> **具体通信链路**:
> ```
> 外部请求 → API Gateway(:8000)
>              ├─ HTTP → LLM Service(:8001)    POST /v1/chat/completions
>              ├─ HTTP → RAG Service(:8003)     POST /search
>              ├─ HTTP → ASR Service(:8005)     POST /v1/transcribe
>              └─ HTTP → TTS Service(:8006)     POST /v1/tts
>
> RAG Service → HTTP → Embedding Service(:8002) POST /v1/embeddings
> RAG Service → HTTP → ChromaDB(:8004)          ChromaDB SDK
> All Service → TCP  → Redis(:6379)             Redis SDK
> ```
>
> **为什么用 HTTP 而不是 gRPC 或消息队列？**
> - gRPC 在 Python 生态不如 HTTP 成熟，调试不方便
> - 消息队列会增加架构复杂度，本项目的并发量（<20 QPS）不需要
> - HTTP 可以用 curl 直接调试，运维友好
>
> **容错机制**:
> - 每个服务客户端有 `health()` 方法，定期检查下游服务状态
> - LLM 调用失败 → 触发 fallback 兜底回复
> - RAG 检索失败 → 降级为纯 LLM 问答（不带知识库上下文）
> - ASR/TTS 失败 → 返回错误但不影响文本问答

## 3. RAG 技术维度

### Q3.1 讲一下 RAG 的完整流程是怎么设计的？

**面试官意图分析**:

这是所有 RAG 项目面试的必考题，面试官用它快速判断你掌握 RAG 的深度：

- **他想干什么 — 金字塔分层判断**：面试官通过你的回答结构来判断你在金字塔的哪一层。① 底层（只会用 LangChain/LlamaIndex 搭积木的）— 开口就是"我用了 LlamaIndex 的 SimpleDirectoryReader..." ② 中层（理解原理但不一定自己写的）— 能讲出索引和检索两个阶段的完整流程 ③ 顶层（自己实现过核心逻辑的）— 除了讲流程，还能说出每个设计决策的"为什么"和"踩过什么坑"。
- **考察要点 ① — 能否区分"离线"和"在线"**：这是 RAG 最基本的思维模型。分不清的人会说"用户提问 → 查向量库 → 生成回答"，漏掉了离线索引这个关键阶段。说出来的人说明对整个生命周期有认知。
- **考察要点 ② — 关键设计决策的"why"比流程本身更重要**：你讲完流程后面试官大概率会追问"为什么先召回多再过滤"、"为什么需要 Re-rank"。如果你能在讲流程时主动带出这些决策理由，会让面试官觉得"这个人思考过"。
- **考察要点 ③ — 是否有工程落地细节**：如果你说"相似度阈值 0.65"、"top_k=3"、"缓存到 Redis TTL 1h"这些具体数字，比泛泛说"做了过滤和缓存"可信度高 10 倍。

**面试官心里的分层判断标准**：
- 听过 RAG 概念 → 能说出"用户问 → 查知识库 → 喂给 LLM"
- 用过 RAG 框架 → 能说出"索引 + 检索 + 组装 Prompt"三阶段
- 真正做过 RAG → 能说出具体的分段策略、向量模型选择、相似度阈值调优、缓存策略
- 深入优化过 RAG → 能说出同义词扩展、混合精排、RAGAS 评估等进阶技术

**参考答案**:

> RAG 分为**离线索引**和**在线检索**两个阶段：
>
> **离线索引阶段**（文档入库）:
> ```
> 原始文档(PDF/Word/TXT/MD)
>   → 文档解析 (pypdf/python-docx/原生)
>   → 文本清洗 (去控制字符/统一换行/去多余空白)
>   → 智能分段 (3 种策略可选)
>   → BGE-M3 向量化 (1024 维)
>   → ChromaDB HNSW 索引入库
> ```
>
> **在线检索阶段**（用户提问）:
> ```
> 用户问题
>   → ① 同义词扩展 (扩展为 2-5 个变体查询)
>   → ② 多路向量检索 (每个变体检索 top_k×2)
>   → ③ 合并去重 (按 doc_id)
>   → ④ 相似度过滤 (cosine similarity ≥ 0.65)
>   → ⑤ 混合精排 (语义×0.6 + 关键词×0.25 + 完整度×0.10 + 权威性×0.05)
>   → ⑥ 返回 Top-3 + 缓存到 Redis
> ```
>
> **关键设计决策**:
> - 为什么先召回多再过滤？检索质量 > 检索速度，保证 recall 的前提下做 precision
> - 为什么需要 Re-rank？Embedding 的相似度排序不够精细，需要引入关键词和文档质量维度的信号

### Q3.2 文档分段有哪几种策略？分别适用于什么场景？

**面试官意图分析**:

这道题考察的是"工程细节"——面试官通过它区分"调过 RAG 库"和"真正落地过 RAG"的人：

- **他想干什么 — 测试你是否踩过"分段没做好的坑"**：凡是在生产环境部署过 RAG 的人都知道，分段策略是整个 RAG 系统中最影响效果但最容易被忽视的环节。面试官想通过这道题确认你不是"用了默认分段"就上线的人。
- **考察要点 ① — 能否说出每种策略的适用场景和副作用**：如果只说"滑动窗口最好"，面试官觉得你只会用默认值。如果你能说出"固定长度可能在句子中间切断"、"语义分段虽然完整但段长度不均匀导致 Embedding 质量不稳定"，说明你在各种策略上都做过尝试和对比。
- **考察要点 ② — 企业特殊处理是最大的区分点**（面试官最关注的部分）：Chunker 的 `_adjust_boundary` 方法（保护国标编号 GB/T、设备型号 SZ11-50000/110、电压值 110.5 kV、章节条款编号）是本题含金量最高的部分。它证明了：① 你真的考虑过行业场景 ② 你不是用通用方案做行业问题 ③ 你有"边界条件处理"的工程习惯。
- **考察要点 ③ — Chunk Overlap 的设置**: 如果你能补充说"50 token 的重叠是为了保证边界信息不会因为切分而丢失，但太大了会增加冗余和存储成本"，说明你对参数选择有成本意识。

**参考答案**:

> 实现了三种分段策略，默认使用滑动窗口：

| 策略 | 原理 | 适用场景 | 优点 | 缺点 |
|------|------|---------|------|------|
| **固定长度** | 每 N 字符切一刀 | 纯文本、无结构文档 | 简单快速 | 可能在句子中间切断 |
| **语义分段** | 按段落/标题自然边界 | 结构化文档（手册、规范） | 语义完整 | 段长度不均匀 |
| **滑动窗口** | 固定大小 + 50 token 重叠 | **通用默认策略** | 平衡召回率和语义完整性 | 存储量大一点 |

> **企业特殊优化**:
> 分段时额外实现了 `_adjust_boundary` 方法，识别并保护以下模式不被切断：
> - 国标编号: `GB/T 12345-2020`
> - 设备型号: `SZ11-50000/110`
> - 电压/功率值: `110.5 kV`, `600 MW`
> - 章节条款编号: `第X章`, `第X条`
>
> 这样保证了"GB/T 12345-2020 标准"作为一个完整的检索单元，不会在中间被切断导致检索不到。

### Q3.3 为什么需要同义词映射？怎么实现的？

这道题考察的核心是"领域自适应能力"——通用 RAG 和行业 RAG 的分水岭：

- **考察要点 ② — 是否理解"多路召回"的价值**：同义词映射不是简单地替换查询词，而是**扩展为多个变体并行检索**。面试官想知道你是否理解"召回率靠多路，精确率靠后续过滤"这个设计模式。如果你能说出"3 个变体各自检索 top_k×2 候选 → 合并去重 → Re-rank"，说明你对召回策略有系统性理解。

> **问题**：企业存在大量术语的同义表达：
> - "变压器" = "主变" = "电力变压器"（同一设备不同叫法）

> **代码关键点**:

### Q3.4 重排序（Re-rank）是怎么做的？为什么 Embedding 排序不够？

**面试官意图分析**:

这道题是区分"初级 RAG 开发者"和"有信息检索功底的人"的关键题：

- **他想干什么 — 测试你是否理解"语义相似度 ≠ 信息相关性"这一核心命题**：这是信息检索领域的经典认知。Embedding 是把文本映射到向量空间，两个向量接近表示"语义相似"，但"语义相似"不等于"这个文档能回答用户的问题"。面试官想看你有没有认识到这个 gap。
- **考察要点 ① — 能否举出反例证明 Embedding 不够**：如果你能举例说明"一篇讲变压器原理的文档和一篇讲变压器巡检的文档，在 Embedding 空间可能很近（都大量使用专业术语），但前者回答不了'巡检项目有哪些'的问题"，面试官会非常满意。这说明你真的观察过检索效果。
- **考察要点 ② — 混合得分公式的权重设计是否有理有据**：面试官会追问"0.6 / 0.25 / 0.10 / 0.05 的权重怎么定的？"如果你说"根据经验调的"，那只是及格。如果你说"在 50 条测试集上做了消融实验，逐步调整权重直到 Top-3 召回率最优"，那是优秀。
- **考察要点 ③ — 为什么不直接用 Cross-encoder 模型做精排？**：如果你主动谈到 Cross-encoder（如 BGE-Reranker），并解释"Cross-encoder 的 query-doc 对送入模型计算量太大，只适用于小候选集，在大候选集下性价比不如混合规则精排"，这证明你研究过不同方案并做了理性取舍。

**权重设置的备选回答方案**（如果面试官追问为什么不是都 0.25）：
> "语义相似度 0.60 是主信号，因为 Embedding 检索本身就靠语义；关键词覆盖 0.25 是对行业场景的纠偏——确保行业核心词命中；文本完整度 0.10 和权威性 0.05 是小幅调整因子，不会主导排序但可以在质量接近时做出区分。"

**参考答案**:

> **Embedding 排序的局限性**:
> 1. 语义相似度 ≠ 信息相关性。两个文本可能在向量空间很近（都包含大量专业术语），但实际回答不了用户的问题
> 2. Embedding 只考虑语义，不考虑文档质量、来源权威性、信息完整度
>
> **本项目的混合精排**:
> ```
> 最终得分 = 语义相似度 × 0.60
>          + 关键词覆盖 × 0.25
>          + 文本完整性 × 0.10
>          + 来源权威性 × 0.05
> ```
>
> | 因子 | 计算方式 | 作用 |
> |------|---------|------|
> | 语义相似度 | Embedding cosine 距离 | 主信号 |
> | 关键词覆盖 | 12 个核心行业关键词（安全/操作/规程/标准...）命中加权 | 提升行业相关性 |
> | 文本完整性 | 完整句子结尾 + 标题结构 + 长度适中(100-1000字) | 优先完整可读的片段 |
> | 来源权威性 | 文件名/标题含"标准""规程""手册""GB""DL/IEC" | 正式文档 > 草稿 |
>
> **为什么不直接用 Cross-encoder？** Cross-encoder 精度更高但速度慢（需要把 query-doc 对送入模型），在候选量大时不实用。先 Embedding 粗排 → 混合精排是性价比最好的方案。

### Q3.5 RAG 检索效果怎么评估？

**面试官意图分析**:

这是检验工程素养的题——面试官在判断你是否具备"数据驱动的迭代能力"：

- **他想干什么 — 区分"做完就完了"和"做完后持续优化"两种工程师**：写一个 RAG 系统不难（网上教程很多），难的是知道怎么衡量它好不好，以及不好的时候怎么优化。面试官通过这道题判断你是否有"评估 → 分析 → 优化 → 再评估"的闭环思维。
- **考察要点 ① — 是否知道 RAG 有标准评估框架**：如果你能提到 RAGAS（Retrieval Augmented Generation Assessment），面试官会认为你不是在闭门造车，而是关注了学术界/工业界的进展。RAGAS 的四个指标——Faithfulness（忠实度）、Answer Relevancy（答案相关性）、Context Precision（上下文精度）、Context Recall（上下文召回率）——是行业标准。
- **考察要点 ② — 测试集的分层设计是否科学**：7 个类别（设备参数、操作流程、安全规范、故障处理、术语解释、标准编号、兜底场景）覆盖了正常场景和 edge case。如果能说明"兜底场景（如'今天天气怎么样'）的目的是测试 fallback 是否正确触发"，说明你的测试设计有针对性。
- **考察要点 ③ — Bad Case 驱动优化的闭环**：最重要的是"发现 bad case → 定位是检索问题还是生成问题 → 针对性修复 → 回归验证"这个流程。如果你能讲出一个具体的 bad case 排查案例，面试官会觉得你的优化经验是真实的。

**参考答案**:

> **评估指标**:
> | 指标 | 定义 | 目标 |
> |------|------|------|
> | Recall@K | Top-K 结果中包含正确答案的比例 | @3 > 90% |
> | MRR | 第一个正确答案的平均排名倒数 | > 0.75 |
> | 答案准确率 | LLM 回复是否正确回答了问题 | > 85% |
>
> **测试集设计**:
> 准备 50 条企业问答对，覆盖 7 个类别：
> 1. 设备参数查询（如"SF6断路器的额定压力？"）
> 2. 操作流程（如"倒闸操作的正确步骤？"）
> 3. 安全规范（如"110kV安全距离？"）
> 4. 故障处理（如"设备着火怎么处理？"）
> 5. 术语解释（如"什么是GIS？"）
> 6. 标准编号（如"GB/T 12345 的内容？"）
> 7. 兜底场景（如"今天天气怎么样？"→ 预期触发 fallback）
>
> **迭代优化闭环**:
> 评估 → 发现 bad case → 分析原因（chunk 没召回？还是 LLM 没回答好？） → 针对性优化（调相似度阈值/加同义词/改 Prompt/调 chunk 大小）

### Q4.1 为什么选 Gemma-3 27B 而不是 Llama-3 或 Qwen？

**面试官意图分析**:

这道题是模型选型决策题，面试官在测试你是否具备"受约束场景下的理性决策能力"：

- **他想干什么 — 判断你选模型的逻辑是"跟风"还是"基于约束做排除法"**：大部分候选人会说"XX 模型效果最好所以我选了它"。但真正在资源受限环境下做过部署的人会说："我先列出硬约束（4×RTX 4090 48GB 总显存），然后排除不符合的方案，然后在剩余选项中挑最优的"。后者才是面试官想听到的回答逻辑。
- **考察要点 ① — 是否以"硬约束"而不是"偏好"做决策起点**：显存是硬约束，中文能力是软约束。如果你说"因为 Gemma-3 中文好所以我选了它"，面试官会追问"Qwen 中文更好你为什么不用？"正确的逻辑是"先排除显存不够的方案（Llama-3 70B 需要 4 卡以上），然后在能跑的方案里比较——27B INT8 量化后每卡 ~18GB，刚好适配 24GB 的 RTX 4090"。

- **考察要点 ③ — 回答中暴露了你是否知道"模型选型不是一次性的"**：如果你补充"模型服务通过 OpenAI 兼容 API 封装，切换到 Qwen 只需要改 vLLM 启动参数和环境变量"，面试官会觉得你有前瞻性设计思维。**

**参考答案**:

| 维度 | Gemma-3 27B | Llama-3 70B | Qwen-2.5 32B |
|------|-------------|-------------|--------------|
| **参数规模** | 27B | 70B | 32B |
| **RTX 4090 适配** | ⭐⭐⭐⭐⭐ 2 卡能跑 | ⭐⭐ 至少 4 卡 | ⭐⭐⭐ 2 卡勉强 |
| **中文能力** | ⭐⭐⭐⭐ 多语言训练 | ⭐⭐⭐ 偏英文 | ⭐⭐⭐⭐⭐ 中文最佳 |
| **开源协议** | Gemma 许可 | Llama 3 社区许可 | Apache 2.0 |
| **vLLM 支持** | ⭐⭐⭐⭐⭐ 原生 | ⭐⭐⭐⭐⭐ 原生 | ⭐⭐⭐⭐⭐ 原生 |
| **vLLM 量化支持** | {AWQ} | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |

> **选择 Gemma-3 的核心原因**:
> 1. **显存约束是硬限制**：RTX 4090 24GB × 2 = 48GB。27B 模型 INT8 量化后 ~14GB + KV Cache ~4GB = 每卡 ~18GB，90% 利用率刚好。32B/70B 模型显存不够或需要激进量化严重影响效果。
> 2. **Gemma-3 是 Google 专门优化的轻量模型**：在 27B 这个参数级别，它的知识密度和多语言能力是顶级的。
> 3. **如果中文场景要求更高**，可以考虑 Qwen-2.5 14B/7B 作为补充或替换。

### Q5.1 4 张 RTX 4090 是怎么分配的？为什么这样分配？

**面试官意图分析**:

这道题是"资源编排能力"的直接考察，面试官用它在验证你是否真正在硬件受限环境下部署过模型：

- **他想干什么 — 测试你是否理解"GPU 分配不是简单的 4 个模型对号入座到 4 张卡"**：这背后有几个隐性约束你必须考虑：① 张量并行需要多张卡通过 NVLink/PCIe 通信 ② 不同模型的使用频率和并发特征不同 ③ 显存留多大 buffer 取决于推理的动态需求。如果候选人说出"一个模型一张卡"，面试官就知道他没有做过真实部署。
- **考察要点 ① — 为什么 Gemma-3 需要 2 卡而不是 1 卡**：这是本题最核心的考察点。面试官想看你是否能算清楚"27B × 2 bytes (FP16) = 54GB > 单卡 24GB → 必须多卡。INT8 量化后权重 14GB + KV Cache 4GB + 激活中间层 4GB ≈ 22GB > 单卡 24GB 但勉强够 → 双卡张量并行更安全"。这个计算过程证明你做过精细的显存预算。
- **考察要点 ② — ASR 和 TTS 为什么共享 GPU 3**：面试官想看你是否考虑了"使用时机"。ASR 和 TTS 在语音对话中不会同时运行（ASR 先跑完后才轮到 TTS），所以可以分时复用同一张卡。如果你能计算出"两个模型各占 ~4GB，即使同时加载也才 8GB < 24GB，但分时使用更安全"，面试官会觉得你考虑全面。
- **考察要点 ③ — Embedding 为什么独占 GPU 2 而不是和 LLM 共享**：如果你说"Embedding 是高频调用的——用户每次提问 + 文档入库都要用它，和 LLM 共享会频繁争抢显存导致 OOM"，这说明你从"工作负载模式"而非"显存大小"来思考资源分配。

```
GPU 0 (24GB): ┐
              ├─ Gemma-3 27B (TP=2)  ~22GB ×2
GPU 1 (24GB): ┘

GPU 2 (24GB): BGE-M3 Embedding       ~6GB

GPU 3 (24GB): ┌─ Whisper-large-v3    ~4GB
              └─ CosyVoice2          ~4GB  (分时使用)
```

> **为什么这样分配**:
> 1. **Gemma-3 独占 2 卡**：27B 参数 INT8 后 ~14GB 权重 + KV Cache ~4GB + 推理中间激活 ~4GB = 每卡 ~22GB。必须双卡张量并行。
> 2. **Embedding 独占 1 卡**：BGE-M3 只需 6GB，但它是检索链路的瓶颈（用户每次提问 + 文档入库都要用），独立部署避免和 LLM 争抢显存导致 OOM。
> 3. **ASR+TTS 共享 1 卡**：两个模型都小（各 ~4GB），且不会同时使用（ASR 先跑完才到 TTS）。共享 GPU 3 节省一卡。

### Q5.2 怎么防止 GPU 显存溢出（OOM）？

**面试官意图分析**:

这道题是"生产环境实战经验"的分水岭——几乎所有在生产环境跑过大模型的人都被 OOM 折磨过：

- **他想干什么 — 判断你是否真正在生产环境踩过 OOM 的坑**：如果你只是跑过 DEMO（单次推理），你不会遇到 OOM。OOM 通常发生在：① 多并发请求时 KV Cache 暴涨 ② 长文本输入导致激活值爆显存 ③ 服务运行一段时间后显存碎片化。面试官想确认你遇到过这些情况并找到了解决方案。
- **考察要点 ① — 是否从"多层防线"的角度来回答**：模型层（量化）+ 推理引擎层（vLLM 参数）+ 调度层（并发控制）+ 部署层（GPU 绑定），四层防线比单点防护更有说服力。面试官想看的是"你在多个层面都做了防御"——这说明你是系统性思考而不是碰巧避开了问题。
- **考察要点 ② — 具体数字比概念更有说服力**：`gpu_memory_utilization 0.90`（而不是默认的 0.95）说明你知道需要预留 buffer；`max_num_seqs 4`（而不是更多）说明你做了并发和显存的权衡计算。这些具体数字暴露了你是真的调过参数而不是看了文档。
- **考察要点 ③ — 是否理解 OOM 的根因分类**：好的回答会区分"模型加载时的 OOM"（权重填不下）和"推理时的 OOM"（KV Cache 动态增长），两者的应对策略不同。前者靠量化，后者靠限制并发序列数和上下文窗口长度。

> 从多个层面做了防御：
>
> **模型层面**:
> - LLM: INT8 AWQ 量化，权重缩减 50%
> - LLM: KV Cache 使用 FP8 精度，减少 KV Cache 显存占用
> - LLM: 上下文窗口限制 ≤ 8192 tokens（而非模型原生的 128K）
>
> **推理引擎层面 (vLLM)**:
> - `--gpu-memory-utilization 0.90`：预留 10% 显存做 buffer
> - `--max-num-seqs 4`：限制同时推理的序列数，避免 KV Cache 膨胀
>
> **调度层面 (API Gateway)**:
> - 最大并发推理请求数 = 4，超过则排队等待
> - 超时降级：推理超过 60 秒返回 fallback
>
> **部署层面**:
> - Docker 容器 `deploy.resources.reservations.devices` 精确绑定 GPU
> - ASR 和 TTS 分时使用 GPU 3，不会同时加载
> - 健康检查 + 异常自动重启

### Q5.3 vLLM 的 Continuous Batching 是怎么回事？

**面试官意图分析**:

这是"推理引擎原理"题，面试官想区分"会用 vLLM"和"理解 vLLM 为什么快"的人：

- **他想干什么 — 测试你对推理优化的理解深度**：vLLM 已经成为 LLM 推理的事实标准，大部分人都知道"vLLM 很快"，但只有少数人能说清楚"它为什么快"。面试官通过这道题判断你的技术好奇心是停留在"会用工具"还是深入到"理解原理"。
- **考察要点 ① — 能否用对比（传统 vs Continuous Batching）的方式解释**：如果你用时间线图对比传统串行推理和 Continuous Batching，面试官会认为你有将复杂概念可视化的沟通能力。即使你不画图，口头描述"传统模式等请求 A 完成才能开始 B，Continuous Batching 在 A 的 decode 间隙插入 B 的 prefill"也足够了。
- **考察要点 ② — 是否知道 PagedAttention 是 Continuous Batching 的基石**：这是核心加分项。PagedAttention 将 KV Cache 分页管理（类似 OS 的虚拟内存分页），允许不同请求的 KV Cache 页动态分配和释放。如果没有 PagedAttention，KV Cache 是连续分配的，请求 A 释放的内存可能被碎片化，无法给请求 B 使用。
- **考察要点 ③ — `max_num_seqs` 参数的工程权衡**：面试官会追问"为什么不把 max_num_seqs 设为 100？"如果你回答"每增加一个并发序列，KV Cache 预分配空间翻倍，4 路并发已经让每张卡使用 ~22GB / 24GB，更多并发会直接 OOM"，面试官会认为你理解参数背后的显存约束。

> 传统 LLM 推理是"一个请求 → 生成完 → 下一个请求"的串行模式。vLLM 的 Continuous Batching 允许**多个请求在 GPU 上同时推理**：
>
> ```
> 传统模式:
> Request A: |████████████████████████████| (3s)
> Request B:                               |████████| (1s)
> Request C:                                        |██████| (1.5s)
> 总耗时: 5.5s
>
> vLLM Continuous Batching:
> Request A: |████████████████████████████|
> Request B:      |████████|                        (在 A 的 prefill 完成后立即插入)
> Request C:                |██████|                 (在 B 完成后立即插入)
> 总耗时: 3.5s (理论加速 35%)
> ```
>
> 核心原理：vLLM 的 PagedAttention 将 KV Cache 分页管理，允许不同请求**动态分配和释放 KV Cache 页面**，一个请求完成后，它的 KV Cache 页被释放，新请求可以立即使用。
>
> 在本项目中，`--max-num-seqs 4` 限制了最多 4 个序列同时推理，避免显存溢出。

## 6. 语音链路维度

### Q6.1 ASR 到 TTS 的完整语音链路是怎么实现的？

**面试官意图分析**:

这道题考察的是"多模态全链路集成能力"和"音视频工程基础"：

- **他想干什么 — 验证你能把三个独立模型串联成一个用户体验无缝的产品**：ASR、LLM、TTS 作为独立模型很多人都用过。但把它们串联成"用户说话 → 系统回话"的完整体验，涉及音频格式转换、时序协调、错误传播处理等工程问题。面试官想看的是"串联能力"，不是对单个模型的了解。
- **考察要点 ① — 是否考虑了前后端音频格式的一致性**：浏览器的 MediaRecorder API 输出 `audio/webm` 格式，ASR 期望 WAV/PCM 格式——如果你不统一这两个格式，语音链路在第一步就会失败。如果你能说出"前端采集 webm → 后端或前端转换为 16kHz mono WAV → ASR"，面试官知道你真的调通过这个链路。
- **考察要点 ② — 是否意识到"语音对话 API 实际上是文本链路 + 音频包装"**：`/voice/chat` 接口的实现是"ASR 转录 → 调用文本问答逻辑 → TTS 合成"，中间的文本链路完全复用。如果你能强调"代码复用，不重复实现 RAG+LLM 逻辑"，说明你有架构设计中的 DRY 意识。
- **考察要点 ③ — 错误传播的处理**：面试官可能追问"ASR 识别错误了怎么办？"如果你回答"ASR 错误会传播到 LLM，LLM 可能回答偏了，但用户通常能通过后续纠正——这是一个多轮对话韧性设计的问题"，说明你考虑了端到端的错误传播。

> 全链路包含 4 个阶段：
>
> **阶段 1: 音频采集**（前端）
> - 浏览器 `navigator.mediaDevices.getUserMedia` API 采集麦克风
> - MediaRecorder API 录制，`audio/webm` 格式
> - 波形动画 + 计时器实时反馈
>
> **阶段 2: 语音识别**（ASR Service）
> - 前端上传音频 → `POST /api/v1/voice/transcribe`
> - Whisper-large-v3 (faster-whisper, float16) 转录
> - VAD 过滤静音，提高有效语音比例
> - 返回文字 + 语言检测结果 + 时间戳分段
>
> **阶段 3: 文本问答**（与文本链路共享）
> - RAG 检索 → LLM 推理（完全复用文本对话链路的代码）
>
> **阶段 4: 语音合成**（TTS Service）
> - API Gateway → `POST /api/v1/tts`
> - CosyVoice2 按句子分块合成（流式）
> - 返回 WAV 音频 → 前端 `<audio>` 自动播放
>
> **语音对语音的完整接口**（黑盒调用）:
> ```bash
> # 一个 API 调用走完 ASR→LLM→TTS 全链路
> curl -X POST /api/v1/voice/chat \
>   -F "audio=@question.wav" \
>   -F "voice_id=default" \
>   --output answer.wav
> ```

### Q6.2 流式 ASR 和非流式 ASR 选哪个？为什么支持两种？

**面试官意图分析**:

面试官在考察"场景驱动的技术选型"：

- **他想干什么 — 判断你是否理解"工程不是选最好的技术，是选最合适的"**：很多年轻工程师会追求"流式"因为觉得更先进。但流式 ASR 的实现复杂度远高于非流式（WebSocket 管理、chunk 时序、部分结果合并），如果场景用不上，就是过度设计。面试官想看你能不能解释清楚取舍。
- **考察要点 ① — 是否基于使用场景来选择默认模式**：当前默认用非流式 REST API，理由是"企业知识问答场景通常是说一句话 → 等回答"。如果你能说明"流式适用于实时会议转写、长篇口述，但本项目不需要"，面试官会认可你的"场景判断力"。
- **考察要点 ② — 为什么保留 WebSocket 流式接口**：这是"预留扩展性"的思路——当前用非流式，但接口层面支持流式，未来如果需要实时对话或长语音场景，不需要改架构。面试官想看到这种"够用就好 + 预留扩展"的工程平衡感。

> 支持两种模式，场景适配：
>
> | 模式 | 接口 | 首字延迟 | 适用场景 |
> |------|------|---------|---------|
> | **非流式** | `POST /v1/transcribe` | 等待完整音频处理完 | 短语音（< 10秒）、移动端 |
> | **流式** | `WS /ws/transcribe` | < 200ms | 长语音、实时对话、边录边识别 |
>
> **非流式实现**: 一次上传完整音频文件，Whisper 整段处理，返回全部结果。简单可靠。
>
> **流式实现**: WebSocket 连接持续接收音频 chunk，累积到足够长度（~64000 bytes，约 2 秒音频）后触发一次识别，逐段返回结果。适合实时对话场景。
>
> 当前默认使用**非流式**（REST API），因为企业知识问答场景通常是"说一句话 → 等回答"，不需要长时实时转录。

## 7. Prompt 工程维度

### Q7.1 System Prompt 是怎么设计的？

**面试官意图分析**:

这道题考察对 LLM 行为的理解深度——Prompt 工程是"软工程"，与算法/系统是互补的：

- **他想干什么 — 判断你是否理解"Prompt 是 LLM 应用程序的核心控制面"**：在 RAG 系统中，即使检索质量很好，如果 Prompt 写得差（如没有明确知识边界），LLM 可能忽略检索到的文档转而使用自己的预训练知识（产生幻觉）。面试官想看你是否意识到 Prompt 设计是 RAG 系统效果的关键一环。
- **考察要点 ① — 6 层分层设计的逻辑是什么**：面试官想听的不是"我把角色设定放在第一层"，而是"为什么这样分层"。如果你能说"LLM 对 Prompt 前半部分关注度更高（Lost in the Middle 现象），所以把最重要的约束——角色和知识边界——放在最前面"，这比单纯描述分层有价值得多。
- **考察要点 ② — "知识边界"是最有含金量的一层**：在 RAG 系统中，最重要的一句话是"只能基于参考文档片段回答，不要使用预训练知识猜测"。这句话直接决定了系统是"基于知识库回答"还是"LLM 自由发挥"。如果你能说清这句话为什么必须写进 System Prompt 而不是放在 User Message 里，面试官会认可你的经验。
- **考察要点 ③ — 安全须知层的企业适配**：如果你说"高压操作相关内容标注 ⚠️ 警告符号，安全风险前置警告"，说明你把行业特殊需求（安全生产是企业的红线）融入到了 Prompt 设计中。这是"行业定制"的微观体现。

> 采用**多层约束结构**，6 个层次逐层收窄 LLM 的输出行为：
>
> ```
> 层次 1: 角色设定 → "你是「智能知识库」—— 企业专业知识问答助手"
> 层次 2: 核心职责 → 基于知识库回答、解释术语、定位规范
> 层次 3: 知识边界 → 只能基于参考文档，不猜测，不知道就说不知道
> 层次 4: 输出格式 → Markdown、分步骤、表格、引用来源
> 层次 5: 安全须知 → 高压操作标 ⚠️，安全风险前置警告
> 层次 6: 语言风格 → 中文、专业、准确、简洁、不模糊
> ```
>
> **为什么分层设计？**
> - LLM 对 Prompt 前半部分的关注度更高（Lost in the Middle 现象）
> - 把最重要的约束（知识边界）放在前面，减少幻觉
> - 格式约束放后面，不影响核心行为

### Q7.2 "不知道"场景的兜底策略是怎么设计的？

**面试官意图分析**:

这道题考察的是"产品思维 + 技术实现的结合"——面试官想看的不仅是技术兜底，更是对用户体验的理解：

- **他想干什么 — 判断你是否意识到"AI 说不知道"是一个产品设计问题而不只是技术问题**：技术上说"我无法回答这个问题"很容易（一行 if 语句），但产品上这会让用户觉得 AI 无能。面试官想看你是不是只从技术角度思考这个场景。
- **考察要点 ① — 两层兜底设计暴露了你对 RAG 链路的理解**：第一层（检索无结果 → 不给 LLM 上下文）和第二层（LLM 异常 → fallback）分别对应不同的故障模式。如果你能清晰区分"检索不到"和"检索到了但 LLM 处理出错"两种场景，面试官会觉得你对系统故障模式有全面认知。
- **考察要点 ② — Fallback 话术的设计考量**："建议您尝试用更具体的关键词重新提问"不是随机写的——它引导用户主动改善查询（因为长具体查询的检索效果更好），体现了"用设计引导用户行为"的产品思维。
- **考察要点 ③ — 是否有"降级可用性"的意识**：fallback 不意味着系统不可用——文本问答还在，其他服务正常。面试官想看到你能区分"部分功能降级"和"系统整体不可用"。

> 实现了两层兜底：
>
> **第一层: RAG 检索层兜底**
> - 相似度阈值 0.65：检索结果最高相似度低于阈值 → 不拼接知识库上下文 → 纯 LLM 回答
> - LLM 的 System Prompt 要求"知识库未找到就说不知道"
>
> **第二层: LLM 输出层兜底**
> - LLM 返回空字符串 → 触发 fallback
> - LLM 服务调用异常 → 触发 fallback
> - Fallback 回复内容:
> ```
> 很抱歉，我在当前知识库中没有找到与您问题相关的信息。
> 建议您：
> 1. 尝试用更具体的关键词重新提问
> 2. 联系相关部门获取最新的技术文档
> 3. 如需人工协助，请联系技术支持团队
> ```
>
> **设计要点**: 不说"我不知道"（显得无能），而是说"知识库未找到 + 给出行动建议"（有用且专业）。

## 8. 对话管理维度

### Q8.1 多轮对话记忆是怎么做的？

**面试官意图分析**:

考察"有状态服务"的设计能力——把这个系统从"单轮兜底回答器"变成"对话助手的核心：

- **他想干什么 — 测试你是否理解"对话记忆的本质是上下文窗口管理"**：对话记忆看起来是存储问题（Redis/数据库），但设计师的关注点是"LLM 的上下文窗口有限，你要把有限的位置留给最重要的信息"。面试官想看你能不能跳出"存储方案选型"的思维定势，进入"上下文窗口经济学"的思考层面。
- **考察要点 ① — "最多保留 10 轮 + 取最近 3 轮拼入 Prompt"的设计决策**：这是本题最关键的考察点。面试官会追问"为什么不是 20 轮？为什么不是 5 轮？"如果你能说"Gemma-3 上下文窗口 8192 tokens，检索结果占 ~1500 tokens，System Prompt ~500 tokens，用户当前问题 ~200 tokens，剩余约 5000 tokens 给历史对话——3 轮对话平均约 1500 tokens/轮，所以取最近 3 轮刚好不超窗口"，这证明了你在做精确的 token 预算而不是拍脑袋。
- **考察要点 ② — Redis + 内存降级的设计**：这不是一个技术难题（Redis 本来就可靠），但主动考虑"如果 Redis 不可用怎么办"体现了防御性设计思维。面试官看到这个会认为你有一种"悲观假设验证"的工程习惯。
- **考察要点 ③ — TTL 24 小时的考量**：为什么不是永久存储？如果你能解释"对话数据不是业务关键数据，过期自动清理避免 Redis 膨胀，同时 24 小时的窗口对一天内的工作连续性足够"，说明你有数据生命周期管理意识。

> **存储方案**: Redis 为主，内存降级
>
> ```python
> # 数据结构
> conv:{conversation_id} → {
>     "title": "对话标题",
>     "messages": [
>         {"role": "user", "content": "...", "timestamp": "..."},
>         {"role": "assistant", "content": "...", "timestamp": "..."},
>         ...
>     ],
>     "created_at": "2026-01-01T00:00:00",
>     "updated_at": "2026-01-01T00:01:00"
> }
> ```
>
> **关键设计**:
> 1. **Redis TTL 24 小时**: 自动过期，无需手动清理
> 2. **历史长度截断**: 最多保留 10 轮（20 条消息），配合 LLM 的 8192 上下文窗口
> 3. **内存降级**: Redis 不可用时自动切换到 Python dict，保证基本可用
> 4. **拼入 Prompt**: 取最近 3 轮历史拼入 LLM 的 messages 数组

### Q8.2 指代消解是怎么实现的？举个具体例子。

**面试官意图分析**:

考察"NLP 实际应用能力"和"对技术方案局限性的诚实认知"：

- **考察要点 ③ — 具体示例的说服力**：如果你能给出"用户说'那上次说的那个呢' → 检测到指代词 + 短问题 → 补全上一轮上下文 → 重新发送给 LLM"的完整链路，面试官会觉得这个功能不是纸上谈兵。

**参考答案**:

> **需要指代消解的场景**:
> - "它的参数是多少？" — 代词"它"
> - "这个设备的维护周期？" — 指示代词"这个"
> - "那上次说的那个呢？" — 省略式提问
>
> **检测逻辑**:
> ```python
> # 检测代词
> pronouns = ["它", "他", "她", "这个", "那个", "这些", "那些", "其"]
> has_pronoun = any(p in question for p in pronouns)
>
> # 检测省略式提问（过短）
> if len(question) < 10:  # 可能是省略式
> ```
>
> **消解策略**:
> ```python
> # 1. 代词消解 - 把上一轮的主题前置
> # 上一轮助手回复: "SZ11-50000/110型变压器的额定容量为50000kVA..."
> # 当前问题: "它的巡检项目有哪些？"
> # 消解后: "关于「SZ11-50000/110型变压器」，它的巡检项目有哪些？"
>
> # 2. 省略消解 - 补充上一轮上下文
> # 上一轮用户问题: "变压器油温异常升高怎么处理？"
> # 当前问题: "原因呢？"（< 10 字）
> # 消解后: "基于之前的讨论（变压器油温异常升高怎么处理？），请回答：原因呢？"
> ```
>
> **局限性**：这是基于规则的消解，不是真正的 Semantic Coreference Resolution。对于跨越多轮的复杂指代（如"第一个方案和第二个方案的区别"），规则方法可能失效。但对于企业内部的知识问答场景（一问一答，逻辑链条短），规则方法足够。

## 9. 部署运维维度

### Q9.1 Docker Compose 怎么管理 GPU 分配和服务依赖？

**面试官意图分析**:

考察"基础设施即代码（IaC）"能力和运维思维：

- **他想干什么 — 判断你对"声明式部署"的理解**：面试官看的不只是你写了 yaml，而是你写的 yaml 是否体现了生产级部署的考量。所有的 Docker 配置——GPU 绑定、健康检查、启动顺序、重启策略——都应该有明确的"为什么"，而不是从网上复制粘贴。
- **考察要点 ① — Docker 健康检查的配置参数体现经验**：`start_period: 120s` 是关键。如果你说"LLM 冷启动需要 2 分钟加载模型，start_period 必须大于这个时间否则容器在模型加载完之前就被判定 unhealthy 反复重启"，面试官知道你真的部署过 LLM 服务。
- **考察要点 ② — depends_on + condition: service_healthy 的坑**：面试官可能追问"你知道 depends_on 只控制启动顺序不等待应用就绪吗？"如果你回答"是的，我们用了 condition: service_healthy 而不是默认的 service_started，确保 ChromaDB 的 /health 接口可访问后才启动 RAG 服务"，面试官知道你踩过这个坑。
- **考察要点 ③ — "为什么不用 K8s"的回答也暴露了经验**：如果你能说出"4 GPU 单机部署用 K8s 是杀鸡用牛刀——K8s 的 Pod 调度、网络策略、存储卷管理的复杂度对于一个单机系统不划算"，面试官会认可你的技术务实态度。

> ```yaml
> # GPU 精确绑定
> deploy:
>   resources:
>     reservations:
>       devices:
>         - driver: nvidia
>           device_ids: ["0", "1"]  # 精确指定用哪几张卡
>           capabilities: [gpu]
>
> # 服务依赖和启动顺序
> depends_on:
>   chromadb:
>     condition: service_healthy  # 等 ChromaDB 健康检查通过
>   embedding-service:
>     condition: service_healthy
>
> # 健康检查配置
> healthcheck:
>   test: ["CMD", "curl", "-f", "http://localhost:8001/health"]

>   timeout: 20s     # 单次超时
>   retries: 3       # 失败 3 次标记 unhealthy

> ```
>
> **为什么用 Docker 而不是 K8s？**
> - 4 GPU 单机部署，不需要 K8s 的集群调度能力
> - Docker Compose 足够满足 GPU 绑定、服务编排、健康检查、自动重启的需求
> - 配置简单，一个 yaml 文件描述全部拓扑

### Q9.2 怎么实现异常自动恢复？

**面试官意图分析**:

考察"生产环境运维意识"和对"故障模式"的认知：

- **他想干什么 — 判断你是否思考过"凌晨 3 点服务挂了怎么办"这个问题**：开发环境不会出现长时间运行导致的问题（如显存碎片化、Redis 连接池耗尽），但生产环境会。面试官通过这道题看你是否思考过"无人值守"场景下的系统容错。
- **考察要点 ① — 三层防护的分工逻辑**：容器层（重启策略）、健康检查层（服务状态检测）、应用层（全链路健康检查）各司其职。如果你能说清楚"容器重启不能解决应用死锁问题但能解决进程崩溃，健康检查能发现应用 hang 住但进程还活着的情况，应用层健康检查能从用户视角评估系统可用性"，面试官会觉得你理解了每层的局限和职责。
- **考察要点 ② — 恢复期间的用户体验考量**：最容易被忽略的点是——LLM 服务重启需要 2 分钟加载模型，这 2 分钟内用户请求怎么办？如果你回答"API Gateway 在 LLM 客户端连接失败时返回 fallback 回复，不阻塞其他服务（文本问答不受影响），服务恢复后自动重新接入"，体现了"在部分服务降级时保持整体可用"的能力。
- **考察要点 ③ — restarts 和 max_attempts 的隐性风险**：面试官可能追问"如果 LLM 服务反复 OOM 重启怎么办？"如果你的 unrest_retries: 5 机制或需要人工介入的告警机制，说明你考虑了"无限重启循环"这个生产环境常见问题。

> 实现了三层防护：
>
> | 层级 | 机制 | 说明 |
> |------|------|------|
> **容器层** | `restart: unless-stopped` | 容器异常退出自动重启 |
> **健康检查** | Docker Healthcheck | 定期检查 /health 接口，不健康则标记并重启 |
> **应用层** | API Gateway 全链路检测 | `GET /api/v1/admin/health` 检查所有下游服务 |
>
> **恢复流程**:
> ```
> LLM 服务 OOM 崩溃
>   → Docker 检测到进程退出
>   → restart: unless-stopped 自动重启
>   → Healthcheck start_period: 120s (等待模型加载)
>   → 健康检查通过 → 重新加入服务
>   → 期间 API Gateway 返回 fallback 回复（不影响其他服务）
> ```

### Q10.1 这个项目最大的技术难点是什么？

**面试官意图分析**:

这是面试中最能拉开差距的题目——面试官通过它判断你在项目中的"核心贡献度"和"技术天花板"：

- **他想干什么 — 找出你在这个项目中的"极限挑战时刻"**：面试官知道，一个人参与过的项目中最难的部分，通常就是他技术能力的上限。如果你说"最难的是配置 Docker"，面试官会判定你的技术深度有限。如果你说"最难的是在 24GB 消费级显卡上跑 27B 大模型还要保持可用延迟"，面试官就知道你触碰过模型部署的核心瓶颈。
- **考察要点 ① — 难点的选择暴露你的技术层级**：面试官在听的不是"难不难"，而是"这个难点在我的认知范围内属于什么水平"。如果你说的难点是"调 RAG 参数比较费时"，面试官会觉得你是入门级 RAG 开发者；如果你说的难点是"消费级 GPU 的显存编排 + 量化精度 trade-off"，面试官会觉得你触碰过系统级优化。
- **考察要点 ② — 是否描述了"尝试 → 失败 → 调整 → 成功"的问题解决过程**：好的回答不是"我们遇到了 X 问题 → 用 Y 方案解决了"，而是"我们最初尝试了 A 方案，因为 XX 原因失败了，然后尝试了 B 方案，发现 YY 问题，最终选择了 C 方案，效果达到 ZZ"。这个迭代过程证明了你的问题解决能力。
- **考察要点 ③ — 是否能量化效果**：如果你说"RAG 召回率从 60% 优化到 88%"，面试官会印象深刻，因为你能用数据说话。每个百分点的提升背后都有具体的技术动作（同义词映射 +15%、阈值调优 +10%、混合精排 +8%、分段优化 +5%），这比笼统地说"做了很多优化"有说服力 100 倍。
- **特别注意**：如果你在 Q1.4 已经说了技术难点，这道题面试官通常会要求你"再展开讲讲其中最有挑战的一个"。你需要准备好一个完整的问题解决故事（STAR 格式：Situation → Task → Action → Result）。

> **难点 1: RTX 4090 消费级显卡跑 27B 大模型**
> - 挑战: 4090 是游戏卡，显存只有 24GB，跑不了 27B 全精度模型（需要 ~54GB）
> - 解决: INT8 量化（权重 → ~14GB）+ 双卡张量并行（每卡 ~18GB）+ KV Cache FP8 + 上下文窗口限制 8192
>
> **难点 2: RAG 检索效果调优**
> - 挑战: 初始版本检索命中率低（~60%），大量召回不相关文档
> - 解决: 多轮迭代 — ① 加入同义词映射（+15% 召回）② 调优相似度阈值（+10%）③ 加入混合精排（+8%）④ 优化分段策略（+5%）最终 ~88% Top-3 召回率
>
> **难点 3: 多模型 GPU 显存编排**
> - 挑战: 4 个模型 + 4 张卡，需要精确分配避免 OOM
> - 解决: Docker GPU 精确绑定 + 模型量化 + vLLM 显存利用率上限 + 推理并发限制

### Q10.2 RAG 检索效果不好的时候，你是怎么排查和优化的？

**面试官意图分析**:

这是"Debug 方法论"题，面试官想知道你的排查逻辑是否系统性：

- **他想干什么 — 判断你是否有一个结构化的排查框架，还是每次靠感觉调参数**：大多数人在 RAG 效果不好时会"改改 chunk size 看看"、"降降相似度阈值试试"。这不是优化，是碰运气。面试官想看的是——你是否有一个"先定位问题归属 → 再缩小范围 → 最后精确打击"的系统化排查流程。
- **考察要点 ① — "先分清是检索的问题还是生成的问题"是这个排查流程的精华**：这是整个方法论中最关键的一步。很多工程师看到 LLM 回答不好就直接去改 Prompt，但其实问题可能出在检索阶段（根本没召回正确的文档）。如果你能说清楚"手动去向量库验证是否命中了相关文档"这个验证步骤，面试官就知道你的排查是有逻辑的。
- **考察要点 ② — 能否给出一个具体的 Bad Case 排查故事**：抽象的排查流程只是骨架，具体的 bad case 才是血肉。用一个真实的 bad case 示例（如"SF6断路器闭锁压力"的排查过程）演示你的排查流程，远比罗列步骤有说服力。
- **考察要点 ③ — 是否有一个可重复的评估循环**："修复 → 在 50 条测试集上回归 → 对比修复前后指标"——这个闭环证明了你的优化不是一锤子买卖，而是可以持续迭代的工程方法。

> 排查流程分三步：
>
> **第一步: 定位问题是检索还是生成**
> ```
> Bad Case: "SF6断路器的闭锁压力是多少？" → LLM 回答"知识库未找到"
> 排查: 手动去向量库搜索 "SF6断路器 闭锁压力"
> 结果: 检索命中了 "SF6断路器额定压力0.6MPa，闭锁压力0.5MPa"
> 结论: 检索是好的，问题在 LLM 没有正确提取
> 修复: 优化 Prompt，要求"先列出所有相关参数，再回答具体问题"
> ```
>
> **第二步: 检索问题 → 分析召回链路的每个环节**
> ```
> 1. 分段粒度是否合适？ ← 检查 chunk 是否切断了关键信息
> 2. 同义词是否覆盖？ ← 搜索词是否在知识库中有其他说法
> 3. Embedding 是否对齐？ ← 查询和文档是否在同一语义空间
> 4. 相似度阈值是否合适？ ← 是否过严/过松
> ```
>
> **第三步: 针对性修复并验证**
> ```
> 修复: 调整 chunk_overlap 50→100 / 添加同义词 / 降低阈值 0.65→0.55 / 增加候选数
> 验证: 在 50 条测试集上重新评估 → 对比修复前后的指标
> ```

### Q10.3 怎么确保知识库答案的准确性？LLM 幻觉问题怎么处理？

**面试官意图分析**:

这道题是 AI 应用面试的"终极题"——几乎每个面试官都会问，因为它同时考察技术深度和诚实度：

- **他想干什么 — 三道合一**：① 考察你对 LLM 幻觉本质的理解（技术深度）② 考察你做了哪些防御措施（工程实践）③ 考察你是否诚实地承认幻觉不能 100% 消除（技术成熟度）。第三点是最容易被忽略但最重要的——面试官不想招一个说"我的系统从不犯错"的不诚实的人。
- **考察要点 ① — "多层防线"的设计思维**：你无法用单一手段解决幻觉问题，需要 Prompt 约束 + 检索过滤 + 输出标注 + 评测监控四层防线。每一层防什么、层与层之间如何互补，是这个回答的核心价值。
- **考察要点 ② — 诚实地说明幻觉的"不可完全消除性"**：面试官特别想听的一句话是"RAG 可以减少幻觉但无法完全消除"。这句话展示了你的诚实和技术成熟度——你知道 LLM 本质是概率生成模型，不是知识库查询引擎。如果你能补充"要进一步消除幻觉，需要引入 RAGAS 的 faithfulness 评测指标，并对敏感回答做人工审核（Human-in-the-loop）"，面试官会觉得你对这个问题有深入研究。
- **考察要点 ③ — "标注来源"这个低成本高收益的措施**：要求 LLM 在回答中标注"参考来源: XX手册.md"是一个简单但极其有效的措施——它让用户可以自己去原文验证，把"信任 AI"转化为"信任可追溯的信息"。这个设计思想（透明化 > 完美化）体现了产品思维。
- **考察要点 ④ — 企业的特殊风险考量**：如果你能补充"对于安全规程类问题，即使 RAG 找到的文档也要谨慎——因为文档本身可能有版本过期的问题，应该在回复中标注文档版本和日期"，面试官会觉得你真的考虑了行业场景的特殊风险。

> **多层防线策略**:
>
> 1. **Prompt 层**: System Prompt 明确要求"只能基于参考文档片段回答，不要使用预训练知识猜测"，并提供 fallback 话术
>
> 2. **检索层**: 相似度阈值过滤（< 0.65 的文档不传入 LLM），减少无关信息干扰
>
> 3. **输出层**: 要求 LLM 在回答后标注信息来源（"参考来源: 变压器维护手册.md"），用户可以追溯验证
>
> 4. **评测层**: 定期在 50 条测试集上评估答案准确率，发现幻觉模式针对性修复 Prompt
>
> **关于彻底的幻觉消除**: RAG 可以减少幻觉但无法完全消除。要进一步提高准确性，需要引入:
> - **RAGAS 评测框架**: 自动评估答案忠实度（faithfulness）
> - **Human-in-the-loop**: 敏感操作（如安全规程）标注"未经验证，请以原文为准"

## 11. 性能优化维度

### Q11.1 系统各环节的延迟分别是多少？怎么优化的？

**面试官意图分析**:

这道题考察"性能优化的系统性思维"——面试官想看的不只是数字，而是你的优化方法论：

- **他想干什么 — 判断你是否理解"优化的本质是找瓶颈"**：很多工程师一上来就全链路优化，没有重点。面试官想看的是——你是否先做了性能分析（profiling → 定位瓶颈 → 排序影响 → 逐点优化），而不是盲目优化。如果你能说"最大瓶颈永远是 LLM 推理（占总延迟 60%+），所以主要精力放在 LLM 优化上"，证明你的优化策略是数据驱动的。
- **考察要点 ① — 优化手段是否基于对延迟来源的深刻理解**：LLM 推理延迟主要来自：① 权重 IO（从显存读取权重的带宽瓶颈）② KV Cache 读写 ③ Attention 计算。INT8 量化缓解了权重 IO 瓶颈（减半显存读取量），FP8 KV Cache 减少了缓存读写带宽，张量并行将 Attention 计算拆分到两张卡——每个优化措施都针对一个具体的延迟来源。如果你能建立这个"延迟来源 → 优化措施"的映射关系，面试官会觉得你有性能分析的底层知识。
- **考察要点 ② — "从哪里优化"比"怎么优化"更体现工程能力**：如果你一上来就说"我们从 6s 优化到 3s"，面试官会追问"你怎么确定优化 LLM 而不是优化 Embedding？"你需要展示 profiling 的思维：先测量各环节延迟 → 发现 LLM 占 60% → 优先优化 LLM → 再优化下一个瓶颈。
- **考察要点 ③ — 优化通常有代价，能否说明 trade-off**：INT8 量化有精度损失（虽然很小），上下文窗口从 128K 限制到 8192 会影响长文档问答。如果能补充"这些 trade-off 在当前场景下是可接受的——企业知识的单次查询不会超过 8000 tokens 上下文"，说明你的优化是场景感知的，不是盲目压缩。

| 环节 | 原始延迟 | 优化后 | 优化手段 |
|------|---------|--------|---------|
| ASR 首字 | ~500ms | **< 200ms** | faster-whisper (CTranslate2) + VAD 过滤静音 |
| Embedding | ~100ms | **< 50ms** | GPU 推理 + 批处理 |
| 向量检索 | ~80ms | **< 30ms** | HNSW 索引 + 检索缓存 |
| LLM 推理 | ~5s | **< 1.5s** | TP=2 + INT8 + FP8 KV Cache + Continuous Batching |
| TTS 首包 | ~1s | **< 300ms** | 流式分句合成 |
| **文本问答 E2E** | ~6s | **< 3s** | 全链路优化 |
| **语音问答 E2E** | ~10s | **< 5s** | 全链路优化 |

> **最大的延迟瓶颈永远是 LLM 推理**，所以主要的优化精力放在了:
> 1. 双卡张量并行（理论加速 ~1.8×）
> 2. INT8 量化（减少显存 IO 带宽压力）
> 3. vLLM Continuous Batching（提升并发吞吐）

### Q11.2 检索缓存是怎么设计的？

**面试官意图分析**:

考察"缓存设计的场景化思维"——面试官想知道你是否考虑了"缓存的收益场景"和"缓存失效策略"：

- **他想干什么 — 判断你是否理解"缓存不是银弹"**：很多工程师会自然地想到"加个 Redis 缓存"，但面试官想知道的是——① 你的缓存命中率高吗（有没有分析过访问模式）② 缓存失效策略合理吗（TTL 为什么是 1 小时不是 1 天）③ 缓存一致性怎么保证（知识库更新时缓存怎么办）。
- **考察要点 ① — 能用"场景驱动"解释缓存的收益**：如果你能说出"企业有很多高频问题（安全距离、巡检项目、操作规程），新员工入职期重复提问很常见——这些场景下缓存的命中率预计 30-50%"，面试官会觉得你的缓存设计是基于业务分析的，不是技术炫技。
- **考察要点 ② — TTL 1 小时 vs 1 天 vs 永久的权衡**：面试官会追问"为什么是 1 小时？"如果你说"1 小时够短——知识库更新后最多 1 小时缓存自动刷新；1 小时够长——高频问题在工作时间内能命中。如果 TTL 过长，知识更新后用户可能看到过时的检索结果"，说明你做了合理的平衡。
- **考察要点 ③ — 两层缓存（Redis + 内存）的降级设计**：Redis 挂了还能用内存缓存，虽然命中率大幅下降（从跨进程共享变为单进程独享），但比完全不缓存好。这种"降级但不断线"的设计反映了你有生产环境的运维意识。

> 两层缓存：
>
> ```python
> # L1: Redis 缓存（跨进程共享）
> key = "retrieval:{query_md5_hash}"
> TTL = 3600  # 1小时
> 
> # L2: 进程内存缓存（Redis 不可用时降级）
> self.cache = {
>     "hash1": [{"text": "...", "score": 0.95}],
>     "hash2": [{"text": "...", "score": 0.88}],
> }
> # LRU 简单实现，超过 1000 条删除最早的
> ```
>
> **什么场景命中率高？**
> - 企业有很多高频问题（"110kV安全距离"、"变压器巡检项目"、"倒闸操作流程"）
> - 新员工入职期重复问同类基础问题
> - 同一个对话中用户追问细节
>
> **缓存失效策略**:
> - TTL 自动过期（1小时）
> - 知识库更新时不主动失效缓存（因为 1 小时 TTL 够短，更新后一小时内自动刷新）

## 12. 安全与合规维度

### Q12.1 怎么保证数据不出企业？

**面试官意图分析**:

考察对"私有化部署"本质的理解深度——这是本项目的核心差异化：

- **他想干什么 — 验证"私有化部署"在你的项目里不是一个空口号**：很多项目都会说"我们支持私有化部署"，但面试官想问的是——你是否真的落实到了每一个技术选型决策中。如果你用 OpenAI 的 Embedding API 但说自己的系统是私有化部署，这就是自相矛盾。
- **考察要点 ① — 是否从"模型 → 存储 → 网络 → 审计"四个维度全覆盖**：面试官想看到的是一个完整的隐私保护矩阵，而不是某个单一措施（如"我们不用云 API"）。你需要证明每个可能的数据出口都被堵住了——所有模型本地推理、所有存储本地化、所有通信内网化、所有日志不外传。
- **考察要点 ② — 与 SaaS 方案的对比是否抓住了"数据主权"这个本质概念**："数据安全"是一个模糊的词，但"数据主权"（Data Sovereignty）是安全合规领域的标准术语。如果你能说"隐私保护不是保护数据不被黑客攻击，更是保护数据不被第三方服务商访问——即使是合法的（如配合训练的条款），企业也不能接受"，面试官会觉得你对隐私有专业级理解。
- **考察要点 ③ — 私有化部署的代价是否被考虑**：面试官可能会追问"私有化部署有什么代价？"如果你能回答"模型更新需要手动下载和部署（不能像 SaaS 一样自动升级）、运维成本增加、需要自有 GPU 硬件"，说明你对方案有全面而非片面认知。

> **从 4 个层面确保**:
>
> | 层面 | 措施 |
> |------|------|
| **网络** | 所有服务在 Docker 内部网络（`kb-net`）通信，仅 API Gateway 和前端对外暴露 |
| **模型** | 全部使用开源模型（Gemma-3/BGE-M3/Whisper/CosyVoice2），本地推理，不调用任何外部 API |
| **存储** | 知识库文档存储在企业服务器本地，ChromaDB 和 Redis 数据卷持久化在本地磁盘 |
| **审计** | 对话日志存储在本地，不发送到外部 |
>
> **与 SaaS 方案的关键区别**:
> - 飞书/钉钉 AI: 企业文档需要上传到字节/阿里的云服务器 → 数据出境风险
> - 本系统: 所有数据停留在企业防火墙内 → **零数据外泄风险**

### Q12.2 如果这是生产环境，还需要补充什么？

**面试官意图分析**:

这是"从原型到生产"的意识测试——面试官通过它判断你的工程成熟度：

- **他想干什么 — 区分"DEMO 级开发者"和"生产级开发者"**：DEMO 的特征是"功能能跑但经不起放大镜看"，生产的特征是"安全、监控、备份、合规都考虑到了"。面试官想看你是否能列出那些"功能开发阶段不需要但生产上线必需的"东西。
- **考察要点 ① — 分类是否全面且逻辑清晰**：如果你能按"安全加固 → 运维监控 → 高可用 → 合规"四个维度来组织回答，而不是散乱地列清单，面试官会觉得你有系统化思维。这说明你不是想到哪说到哪，而是有一个生产环境检查清单的思维框架。
- **考察要点 ② — 是否提到了容易被忽略但重要的点**：例如"对话日志脱敏"——很多开发者会想到安全加固（HTTPS、认证），但很少想到"日志里可能包含敏感信息"。如果你提到"对话内容可能包含商业敏感数据，日志中要脱敏关键词或设置保留期限"，面试官会觉得你的安全意识超越了表面。
- **考察要点 ③ — 等保合规的提及体现了行业意识**：企业（尤其央企国企）的信息系统通常需要通过等保测评。如果你知道等保二级/三级对日志审计、访问控制、数据加密的不同要求，面试官会认可你对行业合规的了解。
- **考察要点 ④ — 回答的"顺序"也体现了优先级意识**：先安全再监控再高可用最后合规——这个顺序本身就是一种优先级判断。安全是第一位的（没安全上线就是事故），监控是第二位的（上线后没有可观测性等于盲飞），高可用是锦上添花。

> **安全加固**:
> - [ ] JWT Token 密钥改为随机生成，对接企业 LDAP/OAuth 单点登录
> - [ ] 前端 + 后端全量 HTTPS（Nginx 反向代理 + Let's Encrypt）
> - [ ] API 限流（FastAPI + slowapi），防止恶意高频调用
> - [ ] 敏感文档权限控制（不同部门只能查自己部门的文档）
>
> **运维监控**:
> - [ ] GPU 显存/温度/功率监控（Prometheus + nvidia-smi exporter + Grafana）
> - [ ] 日志集中收集（ELK / Loki）
> - [ ] 告警规则（OOM、温度 > 80°C、服务不可达）
>
> **高可用**:
> - [ ] Redis 主从 / 哨兵模式
> - [ ] ChromaDB 数据定期备份到对象存储
> - [ ] 模型文件备份
>
> **合规**:
> - [ ] 对话日志脱敏（不记录敏感关键词）
> - [ ] 数据保留策略（超过 X 天的对话日志自动清除）
> - [ ] 等保二级/三级合规评估
